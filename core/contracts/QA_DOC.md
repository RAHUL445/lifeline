# Contract: QA_DOC

Handoff document for a QA engineer for whom the code is a **blackbox**. Generated by the
`handoff-docs` skill at merge. Written to `<artifact_root>/qa_doc.md` from the template
`core/templates/qa_doc.md.tmpl`.

## Required sections (in order)

| # | Section | Source artifacts | Content rule |
|---|---|---|---|
| 1 | Feature flow | spec.md | User action → expected result, step by step. Observable behavior only. |
| 2 | Preconditions & setup | spec.md, test_result.md notes | Environment, accounts, config, test data needed before any case runs. |
| 3 | Test cases | spec.md functional reqs + success criteria | Numbered. Each = steps + expected result. Happy path + edge cases + negative cases. ≥1 case per functional requirement. |
| 4 | Smoke checklist | test_result.md coverage GAPs | Seeded **verbatim** from GAP lines: `⚠ <area> — untested path. Manually exercise: <observable action>`. Translate the file path into the user-visible behavior it backs. |
| 5 | Known limitations / out of scope | spec.md non-goals, review.md notes | What QA should NOT file bugs about. |

## Invariants

- **No implementation detail anywhere**: no file names, function names, class names,
  architecture terms. If a step can't be expressed without naming code, rewrite it as
  observable behavior. (Exception: section 4 may name the gapped file in parentheses as
  a traceability anchor, after the behavioral description.)
- Every functional requirement has at least one test case.
- Every coverage GAP from test_result.md produces exactly one smoke-checklist entry.
- Steps must be executable by someone with the app and this document alone.
