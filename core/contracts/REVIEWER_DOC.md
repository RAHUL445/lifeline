# Contract: REVIEWER_DOC

Handoff document for a human reviewer **who can see the code**. Generated by the
`handoff-docs` skill at merge. Written to `<artifact_root>/reviewer_doc.md` from the
template `core/templates/reviewer_doc.md.tmpl`.

## Required sections (in order)

| # | Section | Source artifacts | Content rule |
|---|---|---|---|
| 1 | Summary | spec.md/bug.md, changelog.md | One paragraph: what changed + why. |
| 2 | Changes by wave/file | changelog.md, task.md | Per file: what changed → why → which spec requirement (or RCA item) it satisfies. Group by wave. |
| 3 | Design decisions | plan.md, task.md notes | Each non-trivial choice + alternatives rejected + why. |
| 4 | Risk areas | review.md, task.md notes | The parts most likely to be wrong; what to scrutinize first. |
| 5 | Review findings & resolutions | review.md, flow.md | Every finding with severity, location, and resolution. Plus the override audit: any `*.skip.user_override` events verbatim. |
| 6 | Test & coverage summary | test_result.md | Suite verdicts per wave; coverage table per changed file; **all GAP lines verbatim**. |

## Invariants

- Code references (paths, symbols) are allowed and encouraged.
- Every spec functional requirement must be reachable from section 2 (a reviewer can
  trace requirement → file → change).
- Every review finding from review.md appears in section 5 — none silently dropped.
- Every coverage GAP line from test_result.md appears in section 6 verbatim.
- Overrides are never hidden: if flow.md has override events, section 5 lists them even
  when everything else is green.
